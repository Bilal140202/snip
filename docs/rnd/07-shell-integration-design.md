## 1. Dynamic Completions

### Problem

The current `snip completions bash/zsh/fish` uses `clap_complete::generate()` to emit **static** completions for `snip`'s own subcommands (`run`, `add`, `rm`, etc.). It does **not** complete snippet names for `snip run <TAB>`.

The user types `snip run buil<TAB>` and gets nothing. This must be fixed: the completion system must read the **current project's** `.snips` file on-the-fly and offer snippet names.

### Architecture

```
User types: snip run buil<TAB>
                          │
                          ▼
           Shell calls completion function
                          │
                          ▼
           Completion function runs:
             snip _complete run "buil" "$PWD"
                          │
                          ▼
           Rust binary: finds .snips, filters by prefix,
           prints one name per line to stdout
                          │
                          ▼
           Shell consumes names, shows menu
```

The key design decision: completions call `snip _complete <subcommand> <partial> <cwd>`. This is a hidden subcommand (prefixed with `_`) that exits fast — it reads one `.snips` file, filters, prints, and exits. Target latency: **< 50ms** for typical projects (1-100 snippets).

### New Rust Subcommand: `snip _complete`

```rust
// src/cli/complete.rs
use anyhow::Result;
use crate::core::snipfile::{find_snipfile, read_snippets};

pub fn run(subcommand: &str, partial: &str) -> Result<()> {
    match subcommand {
        "run" => complete_run(partial),
        "rm" | "edit" => complete_editable(partial),
        _ => Ok(()), // No dynamic completions for other subcommands
    }
}

fn complete_run(partial: &str) -> Result<()> {
    let cwd = std::env::current_dir()?;
    let path = match find_snipfile(Some(&cwd))? {
        Some(p) => p,
        None => return Ok(()), // No .snips — nothing to complete
    };
    let file = read_snippets(&path)?;

    for (key, _) in file.iter() {
        if key.starts_with(partial) {
            println!("{}", key);
        }
    }
    Ok(())
}

fn complete_editable(partial: &str) -> Result<()> {
    // Same as complete_run — both need snippet keys
    complete_run(partial)
}
```

### Bash Completion Script

```bash
# Generated by: snip hook bash  (or: eval "$(snip hook bash)")
# ────────────────────────────────────────────────────────────────
# snip dynamic completions — bash

_snip_completions() {
    local cur prev words cword
    _init_completion || return

    # After 'snip run' or 'snip rm' — complete snippet names
    if [[ ${words[1]} == "run" || ${words[1]} == "rm" ]] && (( cword == 2 )); then
        # Call snip to get names from the CURRENT project's .snips
        local names
        names=$(snip _complete "${words[1]}" "$cur" 2>/dev/null)
        if [[ -n "$names" ]]; then
            # Use compgen to filter against what user typed, add descriptions
            local name desc
            while IFS=$'\t' read -r name desc; do
                COMPREPLY+=($(compgen -W "$name" -- "$cur"))
            done <<< "$names"
            return 0
        fi
    fi

    # Static completions for all other positions
    local subcommands="init add rm edit list run import doctor completions hook"
    if (( cword == 1 )); then
        COMPREPLY=($(compgen -W "$subcommands" -- "$cur"))
        return 0
    fi

    # Subcommand-specific flags
    case "${words[1]}" in
        completions)
            COMPREPLY=($(compgen -W "bash zsh fish elvish powershell" -- "$cur"))
            ;;
        hook)
            COMPREPLY=($(compgen -W "bash zsh fish" -- "$cur"))
            ;;
        import)
            # Complete directory paths
            COMPREPLY=($(compgen -d -- "$cur"))
            ;;
        add)
            # After 'snip add', suggest nothing special (user provides name)
            ;;
    esac
}

# Register the completion function for the 'snip' command
complete -F _snip_completions snip
```

### Zsh Completion Script

```zsh
#compdef snip
# Generated by: snip hook zsh  (or: eval "$(snip hook zsh)")
# ────────────────────────────────────────────────────────────────
# snip dynamic completions — zsh

_snip() {
    local -a subcommands
    subcommands=(
        'init:Create or detect .snips file'
        'add:Add a new snippet'
        'rm:Remove a snippet'
        'edit:Open .snips in $EDITOR'
        'list:List snippets (optionally filtered)'
        'run:Execute a snippet'
        'import:Import snippets from another project'
        'doctor:Validate snippets and report issues'
        'completions:Generate shell completions'
        'hook:Print shell integration code'
    )

    local curcontext="$curcontext" state line
    typeset -A opt_args

    # Define top-level argument parsing
    _arguments -C \
        '1: :->command' \
        '2: :->subarg'

    case "$state" in
        command)
            _describe 'snip subcommand' subcommands
            ;;
        subarg)
            case "$words[1]" in
                run|rm)
                    # Dynamic: call snip to get project snippet names
                    local -a snippet_names
                    snippet_names=(${(f)"$(snip _complete ${words[1]} ${words[CURRENT]} 2>/dev/null)"})
                    if (( $#snippet_names > 0 )); then
                        _describe 'snippet' snippet_names
                    fi
                    ;;
                completions)
                    _values 'shell' bash zsh fish elvish powershell
                    ;;
                hook)
                    _values 'shell' bash zsh fish
                    ;;
                import)
                    _path_files -/
                    ;;
            esac
            ;;
    esac
}

# Register the completion function
_snip "$@"
```

function __fish_snip_complete

    # commandline -ct is the current token being completed
    snip _complete $argv[1] (commandline -ct) 2>/dev/null
end

complete -c snip -n '__fish_seen_subcommand_from rm'   -a '(__fish_snip_complete rm)'

function __fish_snip_no_subcommand
    not __fish_seen_subcommand_from init add rm edit list run import doctor completions hook
end

### Performance Considerations

The `snip _complete` command will be invoked on **every tab press**. It must be fast:

| Concern | Solution |
|---------|----------|
| Cold start of Rust binary (~5-10ms) | Acceptable for tab completion. Cannot avoid. |
| File I/O for `.snips` | `.snips` is typically < 1KB. A single `read_to_string` + TOML parse is < 1ms. |
| Walking parent directories | At most ~10 `is_file()` checks. Negligible. |
| No caching needed | Tab completion is user-initiated (not continuous). One invocation per tab press. |
| Fallthrough | If no `.snips` found, exit immediately with no output. Zero cost. |

**Target budget**: < 20ms wall-clock time for the entire `snip _complete` invocation.

## 2. Shell Hook: Auto-suggest on Typing

### Problem

Tab completion requires the user to **press Tab**. We want something more invisible: as the user types a command that matches a snippet, the full command appears as a grayed-out suggestion (like `zsh-autosuggestions`). The user presses `→` (End/Right arrow) to accept.

### How Atuin Does It (Reference)

Atuin registers a **strategy** with `zsh-autosuggestions`:

```zsh
_zsh_autosuggest_strategy_atuin() {
    suggestion=$(ATUIN_QUERY="$1" atuin search --cmd-only --limit 1 --search-mode prefix 2>/dev/null)
}
ZSH_AUTOSUGGEST_STRATEGY=("atuin" "${ZSH_AUTOSUGGEST_STRATEGY[@]}")
```

This piggybacks on the existing `zsh-autosuggestions` plugin. The plugin calls every registered strategy in order until one returns a non-empty suggestion. Atuin adds itself to the **front** of the strategy list.

### Design for `snip`

#### New Rust Subcommand: `snip _suggest`

```rust
// src/cli/suggest.rs
use anyhow::Result;
use crate::core::snipfile::{find_snipfile, read_snippets};
use crate::core::fuzzy;

/// Print the best snippet command matching the query, or nothing.
///
/// Output: a single line (the command), or nothing (no match).
/// This is called from shell autosuggestion hooks on every keystroke.
pub fn run(query: &str) -> Result<()> {
    if query.is_empty() {
        return Ok(());
    }

    let cwd = std::env::current_dir()?;
    let path = match find_snipfile(Some(&cwd))? {
        Some(p) => p,
        None => return Ok(()),
    };
    let file = read_snippets(&path)?;

    // Strategy 1: Prefix match on snippet key
    //   User types "buil" → matches key "build" → outputs "cargo build"
    for (key, snippet) in file.iter() {
        if key.starts_with(query) {
            // Check the snippet has no placeholders — don't suggest incomplete commands
            if !snippet.has_placeholders() {
                println!("{}", snippet.cmd);
                return Ok(());
            }
        }
    }

    // Strategy 2: Fuzzy match on snippet key
    //   User types "bld" → matches key "build" → outputs "cargo build"
    let all_keys: Vec<String> = file.iter().map(|(k, _)| k.clone()).collect();
    let matches = fuzzy::fuzzy_match(query, &all_keys);
    if let Some(best) = matches.first() {
        if best.score > 100 { // Threshold to avoid low-quality matches
            if let Some(snippet) = file.get(&best.key) {
                if !snippet.has_placeholders() {
                    println!("{}", snippet.cmd);
                    return Ok(());
                }
            }
        }
    }

    // Strategy 3: Prefix match on snippet description
    //   User types "build the" → matches desc "Build the project" → outputs "cargo build"
    let query_lower = query.to_lowercase();
    for (key, snippet) in file.iter() {
        if snippet.desc.to_lowercase().starts_with(&query_lower) {
            if !snippet.has_placeholders() {
                println!("{}", snippet.cmd);
                return Ok(());
            }
        }
    }

    Ok(())
}
```

#### Zsh Integration (with zsh-autosuggestions)

```zsh
# ─── snip autosuggestion strategy for zsh-autosuggestions ───
# Requires: zsh-autosuggestions plugin installed
# This block is emitted by `snip hook zsh`

# Only register if zsh-autosuggestions is available
if typeset -f _zsh_autosuggest_start > /dev/null 2>&1; then
    _zsh_autosuggest_strategy_snip() {
        # $1 = the current buffer content (what the user has typed)
        suggestion=$(snip _suggest "$1" 2>/dev/null)
        # If suggestion is set and is a proper completion of the query,
        # zsh-autosuggestions will display it
        if [[ -n "$suggestion" && "$suggestion" != "$1"* ]]; then
            suggestion=""  # Only suggest if it extends the buffer
        fi
    }

    # Insert snip at the front of the strategy list (highest priority for
    # project-specific snippets). History and other strategies follow.
    if [[ -n "${ZSH_AUTOSUGGEST_STRATEGY:-}" ]]; then
        ZSH_AUTOSUGGEST_STRATEGY=("snip" "${ZSH_AUTOSUGGEST_STRATEGY[@]}")
    else
        ZSH_AUTOSUGGEST_STRATEGY=("snip")
    fi
fi
```

**Key design choice**: `snip` goes **first** in the strategy list because project snippets are more contextually relevant than shell history. If the user is in a Rust project and types `bld`, they want `cargo build`, not a `bld` command from 6 months ago.

#### Bash Integration (via ble.sh)

Bash has no built-in autosuggestion mechanism. The only viable approach is [ble.sh](https://github.com/akinomyoga/ble.sh), a modern bash line editor that supports autosuggestion sources.

```bash
# ─── snip autosuggestion for ble.sh ───
# Requires: ble.sh installed
# This block is emitted by `snip hook bash`

if [[ -n "${BLE_VERSION:-}" ]]; then
    function ble/complete/auto-complete/source:snip {
        local suggestion
        suggestion=$(snip _suggest "$_ble_edit_str" 2>/dev/null)
        # Only suggest if it extends the current buffer
        if [[ -n "$suggestion" && "$suggestion" == "$_ble_edit_str"?* ]]; then
            local remainder="${suggestion:${#_ble_edit_str}}"
            ble/complete/auto-complete/enter h 0 "$remainder" '' "$suggestion"
        fi
    }

    # Add snip as an autosuggestion source
    bleopt complete_auto_source+=snip
fi
```

**For plain bash (no ble.sh)**: No autosuggestion. The user gets tab completion and `Ctrl+S`. This is an acceptable limitation — bash's readline simply doesn't support inline suggestions. Document this clearly.

#### Fish Integration (built-in autosuggestions)

Fish has autosuggestions built in. They are driven by `fish_autosuggest_enable` and functions in `$fish_complete_path`.

```fish
# ─── snip autosuggestion for fish ───
# This block is emitted by `snip hook fish`
# Fish's built-in autosuggestions are history-based. We cannot inject a custom
# source into fish's autosuggestion engine without a patch.
#
# HOWEVER: we can provide a function that the user can bind to a key to
# manually trigger a snip suggestion. This is the "Ctrl+S" approach,
# which is actually better for snip's use case (see Section 3).
#
# For automatic suggestions, fish users should rely on Ctrl+S.
```

**Fish limitation**: Fish's autosuggestion engine only uses command history. There is no plugin API for custom suggestion sources. We accept this and focus the fish experience on `Ctrl+S` + tab completion.

#### Performance: Per-Keystroke Invocation

`snip _suggest` will be called on **every keystroke** when the buffer is non-empty. This is the critical performance constraint:

| Concern | Solution |
|---------|----------|
| Binary cold start | ~5-10ms. Acceptable. |
| File I/O | Same as completions. < 1ms. |
| Fuzzy matching | For < 100 snippets, < 1ms. |
| **Total budget** | **< 15ms per invocation** |
| Debouncing | NOT needed — zsh-autosuggestions already debounces (waits for a pause in typing, ~0.1s). `ble.sh` also debounces. |
| Empty output | If no match, print nothing and exit immediately. ~5ms (just binary startup). |

If 15ms per keystroke is too much for large projects, add a **cache**: `snip _suggest` writes parsed snippets to a temp file (e.g. `/tmp/snip-cache-$INODE`), checks mtime of `.snips` against the cache, and only re-parses if changed. This reduces per-invocation cost to ~5ms (stat + binary start).

## 3. Ctrl+S / Ctrl+O Global Keybinding

### Problem

We need a way for the user to invoke `snip` from **anywhere** in the terminal, not just after typing `snip`. The user presses `Ctrl+S`, a fuzzy picker appears, they select a snippet, and it gets **inserted at the cursor position**.

This is how `snip` becomes muscle memory. The user doesn't need to think "I should use snip." They just press `Ctrl+S` because their fingers learned it.

### How Atuin Does It (Reference)

Atuin's `Ctrl+R` pattern (from the atuin analysis):

1. Shell reads `$BUFFER` (current command line)
2. Passes it as `ATUIN_QUERY=$BUFFER atuin search -i`
3. The `-i` flag = "interactive mode" — TUI takes over the terminal
4. TUI writes to **stderr** (the real terminal), result goes to **stdout**
5. Shell reads stdout and replaces the buffer

The critical trick: **stderr/stdout file descriptor swap**:
```bash
output=$(snip search -i 3>&1 1>&2 2>&3 3>&-)
#    TUI renders to real terminal (fd 2, which is the real stderr)
#    Selected text goes to $output (captured from fd 1, which was swapped)
```

### Design for `snip`

#### New Rust Subcommand: `snip _insert`

This is the "picker mode" that:
1. Reads the current buffer from `SNIP_BUFFER` env var (set by the shell hook)
2. Opens the TUI picker, pre-filtered by the buffer content
3. On selection, prints the selected command to stdout
4. On cancel, prints nothing

The shell hook then replaces the command line buffer with the printed output.

```rust
// src/cli/insert.rs
use anyhow::Result;
use std::env;

use crate::core::snipfile::{find_snipfile, read_snippets};
use crate::ui::picker::{PickerItem, PickerResult, pick};

/// Interactive snippet picker for shell integration.
///
/// Reads SNIP_BUFFER env var for the current command line content.
/// Opens TUI picker. Prints selected command to stdout (or nothing on cancel).
///
/// The TUI renders to stderr (the real terminal).
/// The selected command goes to stdout (captured by the shell).
pub fn run() -> Result<()> {
    let query = env::var("SNIP_BUFFER").unwrap_or_default();
    let cwd = env::current_dir()?;

    let path = match find_snipfile(Some(&cwd))? {
        Some(p) => p,
        None => return Ok(()), // No .snips — do nothing
    };
    let file = read_snippets(&path)?;

    if file.is_empty() {
        return Ok(());
    }

    let items: Vec<PickerItem> = file.iter().map(|(key, snippet)| {
        let display = if snippet.desc.is_empty() {
            format!("{}  {}", key, snippet.cmd)
        } else {
            format!("{}  {}  ({})", key, snippet.desc, snippet.cmd)
        };
        PickerItem {
            key: snippet.cmd.clone(),
            display,
            detail: snippet.desc.clone(),
        }
    }).collect();

    match pick(&items)? {
        PickerResult::Selected(cmd) => {
            // Print ONLY the command, no extra text
            // This goes to stdout, captured by the shell
            print!("{}", cmd);
            Ok(())
        }
        PickerResult::Cancelled => Ok(()),
    }
}
```

#### Important: The picker must write the TUI to stderr

The existing `picker.rs` uses `io::stdout()` for rendering. For the shell integration, we need to render to **stderr** instead. This requires a small refactor:

```rust
// In picker.rs — add a parameter or use an env var
fn render_picker_ui(query: &str, items: &[&PickerItem]) {
    // When called from shell integration (SNIP_BUFFER is set),
    // render to stderr instead of stdout
    let use_stderr = std::env::var("SNIP_BUFFER").is_ok();
    let mut writer: Box<dyn Write> = if use_stderr {
        Box::new(io::stderr().lock())
    } else {
        Box::new(io::stdout().lock())
    };
    // ... render using `writer` instead of `stdout` ...
}
```

Similarly, `enable_raw_mode()` and crossterm events must work on the **terminal** (which is stderr when fd-swapped). This is actually the default behavior since crossterm uses `/dev/tty` internally.

#### Zsh Keybinding

```zsh
# ─── snip Ctrl+S keybinding for zsh ───
# This block is emitted by `snip hook zsh`

# Create a zle widget
zle -N snip-picker _snip_picker_widget

# Bind to Ctrl+S (also Ctrl+O as alternative)
# Note: Ctrl+S is traditionally XOFF (flow control). We disable it.
stty -ixon 2>/dev/null  # Disable flow control so Ctrl+S works
bindkey '^S' snip-picker
bindkey '^O' snip-picker

_snip_picker_widget() {
    emulate -L zsh
    zle -I  # Invalidate the line editor display

    # Capture the current buffer content as the query
    local output
    output=$(SNIP_BUFFER="$BUFFER" snip _insert 3>&1 1>&2 2>&3 3>&-)

    # Restore the terminal state (bracketed paste, etc.)
    zle reset-prompt
    echo -n "${zle_bracketed_paste[1]}" >/dev/tty

    if [[ -n "$output" ]]; then
        # Replace the ENTIRE buffer with the selected command
        BUFFER="$output"
        CURSOR=${#BUFFER}

        # If output starts with magic prefix, also execute
        if [[ "$BUFFER" == __snip_execute__:* ]]; then
            BUFFER="${BUFFER#__snip_execute__:}"
            zle accept-line
        fi
    fi
}
```

**Dual mode**: Enter selects and inserts. Ctrl+Enter (or a toggle in the picker) selects, inserts, AND executes immediately. This is signaled by the `__snip_execute__:` prefix.

#### Bash Keybinding

```bash
# ─── snip Ctrl+S keybinding for bash ───
# This block is emitted by `snip hook bash`

# Disable XOFF so Ctrl+S works
stty -ixon 2>/dev/null

# Bash readline binding
# We use bind -x to execute a shell function that launches the picker
_snip_picker_bash() {
    # Save the current readline buffer
    local current_line="$READLINE_LINE"
    local current_point="$READLINE_POINT"

    # Launch the picker with the current buffer as query
    local output
    output=$(SNIP_BUFFER="$current_line" snip _insert 3>&1 1>&2 2>&3 3>&-)

    if [[ -n "$output" ]]; then
        # Check for auto-execute prefix
        if [[ "$output" == __snip_execute__:* ]]; then
            output="${output#__snip_execute__:}"
            READLINE_LINE="$output"
            READLINE_POINT=${#output}
            # Accept the line (execute it)
            # In bash bind -x, we can't directly call accept-line,
            # so we print the line and let it be submitted
        else
            READLINE_LINE="$output"
            READLINE_POINT=${#output}
        fi
    fi
}

# Bind Ctrl+S and Ctrl+O
bind -x '"\C-s": _snip_picker_bash'
bind -x '"\C-o": _snip_picker_bash'
```

**Bash limitation**: `bind -x` with `READLINE_LINE`/`READLINE_POINT` is available in Bash 4.3+. For older bash, the picker still works but requires the atuin-style macro dispatch (see atuin analysis, Section A, Layer 2). For v1, we target Bash 4.3+ and document the requirement.

#### Fish Keybinding

```fish
# ─── snip Ctrl+S keybinding for fish ───
# This block is emitted by `snip hook fish`

function _snip_picker_fish
    # Capture current command line
    set -l current (commandline -b)
    set -l output (SNIP_BUFFER="$current" snip _insert 3>&1 1>&2 2>&3 3>&-)

    if test -n "$output"
        # Check for auto-execute prefix
        if string match --quiet '__snip_execute__:*' "$output"
            set output (string replace "__snip_execute__:" "" -- "$output")
            commandline -r "$output"
            commandline -f execute
        else
            commandline -r "$output"
        end
    end
end

# Bind Ctrl+S and Ctrl+O
bind \cs _snip_picker_fish
bind \co _snip_picker_fish
```

#### The Insert vs. Execute Distinction

The picker TUI needs two accept modes:

| Key | Action | Shell Output |
|-----|--------|-------------|
| `Enter` | Insert at cursor | `docker compose up -d` |
| `Ctrl+Enter` | Insert AND execute | `__snip_execute__:docker compose up -d` |

The shell hook strips the prefix and calls `accept-line` / `commandline -f execute`.

### Ctrl+S vs. Flow Control (XOFF)

`Ctrl+S` is traditionally the XOFF character (pauses terminal output). We must disable it:

| Shell | Method |
|-------|--------|
| zsh | `stty -ixon` in the hook script |
| bash | `stty -ixon` in the hook script |
| fish | Fish ignores `ixon` by default, but we add `stty -ixon` for safety |

We also bind `Ctrl+O` as an alternative for users who can't disable flow control (e.g., serial console users).

### Fallback: What If the TUI Can't Start?

If `snip _insert` detects that stdin is not a terminal (piped, non-interactive), it should fall back to a simple line-based selection:

```rust
pub fn run() -> Result<()> {
    let is_tty = crossterm::tty::IsTty::is_tty(&std::io::stdin());

    if !is_tty {
        // Non-interactive fallback: just list snippets
        // The shell hook gets nothing and does nothing
        return Ok(());
    }

    // ... TUI picker as above ...
}
```

## 4. `eval "$(snip hook)"` Unified Entry Point

### The Single Line

```bash
# Add to .bashrc / .zshrc / config.fish:
eval "$(snip hook)"          # auto-detects shell from $SHELL
# or explicitly:
eval "$(snip hook zsh)"
eval "$(snip hook bash)"
eval "$(snip hook fish)"
```

### What `snip hook` Outputs

The `snip hook` command prints shell code to stdout. The `eval` executes it in the current shell session. This single command sets up **everything**:

| Component | What It Sets Up |
|-----------|----------------|
| Tab completions | `_snip_completions` / `_snip` / `__fish_snip_complete` |
| Ctrl+S keybinding | `snip-picker` widget (zsh), `bind -x` (bash), `bind \cs` (fish) |
| Auto-suggestions | `_zsh_autosuggest_strategy_snip` (zsh + zsh-autosuggestions), `ble/complete/auto-complete/source:snip` (bash + ble.sh) |
| Flow control disable | `stty -ixon` |

### New Rust Subcommand: `snip hook`

```rust
// src/cli/hook.rs
use clap::{Args, ValueEnum};
use anyhow::Result;

#[derive(Debug, Args)]
pub struct HookCmd {
    /// Shell to generate hook script for. Auto-detected if omitted.
    pub shell: Option<HookShell>,
}

#[derive(Debug, Clone, ValueEnum)]
pub enum HookShell {
    Bash,
    Zsh,
    Fish,
}

impl HookCmd {
    pub fn run(&self) -> Result<()> {
        let shell = match &self.shell {
            Some(s) => s.clone(),
            None => detect_shell()?,
        };

        let script = match shell {
            HookShell::Bash => include_str!("../../shell/hook.bash"),
            HookShell::Zsh => include_str!("../../shell/hook.zsh"),
            HookShell::Fish => include_str!("../../shell/hook.fish"),
        };

        print!("{}", script);
        Ok(())
    }
}

fn detect_shell() -> Result<HookShell> {
    let shell = std::env::var("SHELL").unwrap_or_default();
    if shell.contains("zsh") {
        Ok(HookShell::Zsh)
    } else if shell.contains("fish") {
        Ok(HookShell::Fish)
    } else {
        Ok(HookShell::Bash)
    }
}
```

_snip_completions() {
    local cur prev words cword
    _init_completion || return

    if [[ ${words[1]} == "run" || ${words[1]} == "rm" ]] && (( cword == 2 )); then
        local names
        names=$(snip _complete "${words[1]}" "$cur" 2>/dev/null)
        if [[ -n "$names" ]]; then
            COMPREPLY=($(compgen -W "$names" -- "$cur"))
            return 0
        fi
    fi

    local subcommands="init add rm edit list run import doctor completions hook"
    if (( cword == 1 )); then
        COMPREPLY=($(compgen -W "$subcommands" -- "$cur"))
        return 0
    fi

    case "${words[1]}" in
        completions) COMPREPLY=($(compgen -W "bash zsh fish elvish powershell" -- "$cur")) ;;
        hook)        COMPREPLY=($(compgen -W "bash zsh fish" -- "$cur")) ;;
        import)      COMPREPLY=($(compgen -d -- "$cur")) ;;
    esac
}
complete -F _snip_completions snip

# ─── Ctrl+S / Ctrl+O Keybinding ──────────────────────────────────

    output=$(SNIP_BUFFER="$current_line" snip _insert 3>&1 1>&2 2>&3 3>&-)

            READLINE_LINE="$output"
            READLINE_POINT=${#output}

            READLINE_LINE="$output"
            READLINE_POINT=${#output}

    fi
}

#!/bin/zsh
# ────────────────────────────────────────────────────────────────────
# snip shell integration — zsh
# Generated by: eval "$(snip hook zsh)"
# ────────────────────────────────────────────────────────────────────
# Disable XOFF flow control so Ctrl+S is available
stty -ixon 2>/dev/null

# ─── Tab Completions ──────────────────────────────────────────────
_snip() {
    local -a subcommands
    subcommands=(
        'init:Create or detect .snips file'
        'add:Add a new snippet'
        'rm:Remove a snippet'
        'edit:Open .snips in $EDITOR'
        'list:List snippets (optionally filtered)'
        'run:Execute a snippet'
        'import:Import snippets from another project'
        'doctor:Validate snippets and report issues'
        'completions:Generate shell completions'
        'hook:Print shell integration code'
    )

    local curcontext="$curcontext" state line
    typeset -A opt_args

    _arguments -C \
        '1: :->command' \
        '2: :->subarg'

    case "$state" in
        command)
            _describe 'snip subcommand' subcommands
            ;;
        subarg)
            case "$words[1]" in
                run|rm)
                    local -a snippet_names
                    snippet_names=(${(f)"$(snip _complete ${words[1]} ${words[CURRENT]} 2>/dev/null)"})
                    if (( $#snippet_names > 0 )); then
                        _describe 'snippet' snippet_names
                    fi
                    ;;
                completions) _values 'shell' bash zsh fish elvish powershell ;;
                hook)        _values 'shell' bash zsh fish ;;
                import)      _path_files -/ ;;
            esac
            ;;
    esac
}
compdef _snip snip

zle -N snip-picker _snip_picker_widget
bindkey '^S' snip-picker
bindkey '^O' snip-picker

    output=$(SNIP_BUFFER="$BUFFER" snip _insert 3>&1 1>&2 2>&3 3>&-)
    zle reset-prompt

        CURSOR=${#BUFFER}
        if [[ "$BUFFER" == __snip_execute__:* ]]; then

        fi
    fi
}

Users can disable parts of the hook via environment variables:

These are checked **in the shell script** (not in Rust), so they take effect immediately without recompilation:

## Appendix A: Flow Control (XOFF) Considerations

`Ctrl+S` sends XOFF (0x13, DC3), which tells the terminal driver to stop sending output. This can hang the terminal. Our hook scripts run `stty -ixon` to disable this.

However, there are cases where the user might want XOFF:
- Serial console connections
- Some terminal multiplexers
- `screen` sessions

**Mitigation**: Always bind `Ctrl+O` as an alternative. `Ctrl+O` is unused in most shells and doesn't conflict with anything. Document: "Use Ctrl+O if Ctrl+S doesn't work for you."

## Appendix B: Comparison with Static `snip completions`

The existing `snip completions bash/zsh/fish` generates static completions using `clap_complete`. This remains useful for:
- CI environments where the `.snips` file might not be present
- Users who don't want the full `snip hook` integration
- Systems where dynamic completion (calling `snip` on every tab) is too slow

The `snip hook` approach **replaces** `snip completions` for interactive use. Keep `snip completions` as a fallback.

## Appendix C: Security Considerations

The `eval "$(snip hook)"` pattern is a standard shell integration approach used by `atuin`, `rustup`, `direnv`, `mise`, `starship`, and many others. The security model is:

1. `snip hook` is a local binary — no network calls
2. The output is a shell script that only defines functions and bindings
3. No arbitrary code execution beyond what the user's `.bashrc`/`.zshrc` already runs
4. The hidden `_complete`, `_suggest`, `_insert` subcommands only read `.snips` files and print to stdout

No elevated privileges are needed. No network access. No file writes.
